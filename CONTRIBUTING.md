# Contributing to K-ATOM

Thank you for your interest in contributing to **K-ATOM**.

K-ATOM is not just a software project — it introduces a **new knowledge primitive**.
Contributions are welcome, but they must respect the conceptual integrity of the project.

This document explains **how** and **why** to contribute.

---

## What K-ATOM is (and is not)

K-ATOM defines the **atomic unit of knowledge**:
a self-contained, verifiable object that is simultaneously:
- human-readable
- machine-executable
- portable as a single artifact

K-ATOM is **not**:
- a file format experiment
- a compression trick
- a steganography project
- a UI-only demo

Contributions should reinforce K-ATOM as a **primitive**, not dilute it into a tool or gimmick.

---

## Types of contributions welcome

We welcome contributions in the following categories:

### 1. Core functionality
- Improvements to Maker or Viewer
- Better SVG/SVGZ handling
- Performance or size optimizations
- Deterministic behavior guarantees

### 2. Verification & integrity
- Cryptographic improvements
- Signature workflows
- Provenance and identity mechanisms
- Trust-chain experiments

### 3. Agent & AI integration
- Agent ingestion patterns
- RAG / graph / vector integrations
- Orchestration examples
- Memory and context workflows

### 4. Documentation & examples
- Clear usage examples
- New real-world use cases
- Integration guides (e.g. GraphDBs, AI pipelines)

---

## Conceptual integrity guidelines

When contributing, please ensure that:

- A **K-ATOM remains atomic**
  - No partial loading that breaks integrity
  - No hidden external dependencies
- A **K-ATOM remains self-contained**
  - All required knowledge is inside the artifact
- **Human + AI symmetry is preserved**
  - If machines can read it, humans should be able to inspect it
- **Visual identity remains meaningful**
  - Visuals should encode identity, not decoration

If a contribution weakens these principles, it is unlikely to be accepted.

---

## How to contribute

1. Fork the repository
2. Create a feature branch
3. Make your changes
4. Include a clear explanation of:
   - what problem this solves
   - how it aligns with K-ATOM as a primitive
5. Open a pull request

Small, focused PRs are preferred.

---

## Attribution & continuity

If your contribution introduces:
- a new concept
- a new abstraction
- or a significant extension

please document it clearly so it can be cited, discussed, and evolved without fragmentation.
